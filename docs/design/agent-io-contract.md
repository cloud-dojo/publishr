# Publishr エージェントI/O契約（実装直結スペック）

> 📑 全体の目次・真実源マップは [正本マップ](../README.md)／未確定論点は [open-issues.md](../planning/open-issues.md)。

> **位置づけ**: 企画と実装の境界を固める唯一の実装直結ドキュメント。各STEPのエージェントが「何を受け取り・何を返すか」（JSONスキーマ）と「どう判断するか」（プロンプト骨子）を定義する。**W2のE2E縦通し**で友人（エンジニア）が手を動かす単位がこれ。
> **原典との関係**: パイプライン構造＝`技術アーキテクチャ.md` §1、データモデル＝同 §3、スコープ/DoD＝`MVPスコープ.md` §3、Eval＝同 §9、役割設計＝構想サマリー §6-2d。本書はそれらを**実装可能な契約**に翻訳したもの。新しい仕様判断はせず、既存確定事項を機械可読に落とす。
> **スコープ**: MVP（実働5週間版）。3ソース（Drive＋Calendar＋Tasks）＋初期登録プロフィール／STEP1=1エージェント圧縮・週1回（土朝）／STEP2=3階層構造（企画リーダー＝スコアゲート／企画担当者／サブ実行者）＋スコア閾値の差し戻し（最高3ラウンド）／STEP3著者はテーマ確定時に**都度生成・各run4人**（固定プール選抜から変更）。**【2026-06-18 MTG改定】トリガーは週3回×4冊/回=週12冊（本命2回×4冊=週8冊＋日曜セレンディピティ1回×4冊）。保持期間=過去4週間（最大約48冊=4冊×3回×4週）。動的フィルタリング: 書庫移動済みの本は入荷一覧から非表示（I-29/I-30参照）。旧: 土/水=各5冊・日=セレンディピティ5冊=週15冊・7日保持**
> **ステータス**: ✅ MTG 2026-06-05で関連論点（観測束保存先I-19・読書ログ集約I-9・Drive Pickerフォルダ単位G1-13・連携G1-3）を確定し反映済み。
> **最終更新**: 2026年6月3日

## 0. 読み方・記法

- スキーマは説明用の擬似JSON。`// コメント`で補足、`?`付きフィールドは任意。実装時はPydantic（Python/ADK側）かTypeScript型に落とす。
- **「エージェント」＝判断する所だけ**。データを動かすだけの処理は「ツール（バッチ）」と明記し、エージェントにしない（構想 §6-2d 設計規律）。
- 各契約に **使用モデル**（Flash主体／難判定のみPro）を付す。コスト規律（アーキ §11 R7）。
- Firestoreのコレクションは `技術アーキテクチャ.md` §3 に準拠：`plans/` `books/` `personas/`（`ephemeral`/`fromFavorite` を追加）`users/`（`initialProfile`・`favoriteAuthors[]` を追加）。

## 2. STEP0：観測（ツール・非エージェント）

**役割**: 3ソースのテキストを取得し、判断せずに束ねる。決定論的処理＝ツール。
**使用モデル**: なし（API呼び出し＋テキスト抽出のみ）。

### 入力
`users/{userId}.connectedSources`（アーキ §3）。MVPは `drive` / `calendar` / `tasks` の3つ。**Drive は Google Picker で選んだ `folderIds[]`（フォルダ単位）配下のみ取得**（drive.fileは走査不可・G1-13＝MTG 2026-06-05確定。フロント=Picker UI(鉄田)／バック=ID保存・読取(一瀬)）。

### 出力：`ObservationBundle`
```jsonc
{
  "userId": "string",
  "collectedAt": "2026-06-02T06:00:00Z",   // ISO8601
  "drive": {
    "files": [
      {
        "fileId": "string",
        "name": "string",
        "mimeType": "application/vnd...",
        "folderLabel": "業務",               // 業務 | 趣味（初回にユーザーが付与・アーキ§3 labels[]）
        "textExcerpt": "抽出テキスト（上限 例:4000字）",
        "modifiedTime": "ISO8601"
      }
    ]
  },
  "calendar": {
    "events": [
      {
        "title": "string",
        "start": "ISO8601",
        "end": "ISO8601",
        "attendeesCount": 5,                 // 参加人数（多い=重要局面の手がかり）
        "recurring": true                    // 定例か単発か
      }
    ]
  },
  "tasks": {
    "items": [
      { "title": "string", "due": "ISO8601?", "status": "needsAction|completed", "notes": "string?" }
    ]
  },
  "readingFB": {                              // 2サイクル目以降。初回は空配列
    "highlights": [ { "bookId": "string", "text": "string", "createdAt": "ISO8601" } ],  // 取得元＝books/{bookId}/highlights サブコレクション（I-9）
    "feedback":   [ { "bookId": "string", "rating": 1, "wantsSequel": false, "readPercent": 0.0, "dropped": false } ]  // 取得元＝books/{bookId}.feedback 集約（I-9）。dwellSecはMVP対象外
  }
}
```

> **保存先（I-19＝MTG 2026-06-05確定）**: STEP0が吐く `ObservationBundle` は **`users/{uid}/observations/{YYYY-MM-DD}` サブコレクション**（日付docID＝冪等・再runは上書き）に**フル束をインライン保存**（STEP1再実行時にGoogle API再叩き不要＝デモ再現性 I-14）。サーバ専用書込・本人read（`Firestoreセキュリティルール.md` §3）。生テキストのGCS分離・retention/TTLは将来F項目。
> **読書FBの取得元（I-9＝MTG 2026-06-05確定）**: `readingFB.highlights` は `books/{bookId}/highlights` サブコレクション、`readingFB.feedback` は `books/{bookId}.feedback`（`{rating,wantsSequel,readPercent,dropped}`）集約から読む。滞在ログ(dwellSec)はMVP対象外。
> **取得範囲（2026-06-03 v2・G1-14解決）**: **Calendar/Tasks は ±14日（過去2週間＋先2週間）・生データのみ（要約しない）**。「最近何をしたか（過去）」と「何を控えるか（未来＝例6/5役員報告）」の両方を拾う。Tasks は**未完了＋直近の完了**。Drive の `textExcerpt` は**1ファイル上限 約4,000字**でトリム（冒頭＋見出し優先）。3ソース疎通はW1で確認（MTG 2026-06-05合意）。
> **注意**: Gmail本文は読まない（MVPはGmail自体スコープ外）。MVPは**キュレーション済みサンプルDrive 約10件**が対象なので、4,000字×10件≒3万字≒約2万トークンで余裕（コスト・コンテキスト対策／§10-6確定）。乱雑な実Drive接続時のトリム精緻化は将来課題。`folderLabel` で業務/趣味を読み分ける（構想 §6-5）。**STEP0は生データのみを返し、要約・解釈はしない**（解釈はSTEP1に一本化）。

## 3. STEP1：読者分析エージェント（×1）

**役割**: 観測束から「いまのこの人」を**3層構造**のプロファイルに統合。**安定情報は再分析せず保持し、週次で変わる現状情報・読書行動を頑張って分析する**（2026-06-03 v2）。
**使用モデル**: **Gemini Pro**（全工程の起点＝品質が全体にカスケードするため。ハイブリッド方針の中でここはProに上げる）。
**実行頻度**: **週1回（土朝のみ）**。読者の状況は3日では大きく変わらないため、1週間分の差分を土朝に一括分析し、生成した `profile` を水・日の企画runで再利用する。
**Firestore**: 出力を `users/{userId}.profile`（`{base, currentWork, readingBehavior}` のサブ構造）に保存。`base` は更新頻度が低い。

### 入力
- `ObservationBundle`（STEP0出力）
- `users/{userId}.profile`（前回プロファイル・初回はnull）
- `users/{userId}.initialProfile`（登録時の初期入力＝業界/職種/役職/最近の関心/読書傾向。`base` の主要シグナル）

### 出力：`ReaderProfile`（3層構造）
```jsonc
{
  "userId": "string",
  "generatedAt": "ISO8601",

  // ① base：安定・ほぼ固定。initialProfile由来＋初回だけDrive補完。再分析しない＝保持
  "base": {
    "industry": "食品・飲料メーカー",
    "jobType": "マーケティング・ブランド",
    "position": "課長・マネージャー（2026/04新任）",
    "orgScale": "部下7名（年上のベテラン佐藤42歳を含む）",  // ★initialProfileに無い→初回にDriveから一度抽出し以後保持
    "readingGenres": ["すぐ使える実践書・ハウツー", "事例・ストーリーで学ぶ"]
  },

  // ② currentWork：週次更新・★分析の主戦場（Drive＋Calendar＋Tasks）
  "currentWork": {
    "currentSituation": "新任2ヶ月。年上部下の任せ方に悩み、初の評価面談と6/5役員報告を控える移行期",
    "activeWorkThemes": ["ブランドリニューアルの意思決定", "新任マネジメント"],
    "challenges": ["年上の実力者部下にどこまで任せるかの距離感", "抱え込みと丸投げのあいだで揺れる"],  // ★仕事の課題・悩みの所在
    "upcomingKeyEvents": [ { "title": "役員中間報告", "date": "2026-06-05" } ],  // Calendar由来・控える重要局面
    "evidence": [
      { "claim": "年上の部下の任せ方に悩む", "source": "drive:06_1on1_議事メモ集" },
      { "claim": "6/5に役員中間報告を控える", "source": "calendar:06-05_役員中間報告" }
    ]
  },

  // ③ readingBehavior：週次更新・学習シグナル（readingFB由来・初回は空）
  "readingBehavior": {
    "recentReads": [ { "title": "（既読本）", "theme": "1on1" } ],  // ★既読傾向＝次企画の被り回避
    "highlightsSummary": "「期待役割の合意」に強く反応",                // 最高純度の関心シグナル
    "dropSignals": [ { "title": "（途中離脱本）", "reason": "一般論すぎ" } ],  // ミスマッチ
    "feedbackSummary": "実務直結を高評価／抽象論は低評価",
    "serendipityTolerance": "mid",   // low|mid|high（readingGenres初期値＋行動で調整）
    "stylePreference": "実務的・対話的"  // 好む文体（行動から推定）
  }
}
```

### プロンプト骨子
```
あなたは読者分析の専任エージェント。出力は3層構造（base / currentWork / readingBehavior）。
- ① base（業界/職種/役職/組織規模/読書傾向）は安定情報。initialProfile をそのまま載せ、再分析しない。
  ただし orgScale（部下人数等）が未確定なら初回だけ Drive（チーム体制表等）から一度抽出して確定する。
- ② currentWork は今週の観測（Drive/Calendar/Tasks）から「いまの局面・取り組む仕事・課題・控える重要局面」を分析。
  challenges（悩みの所在）と upcomingKeyEvents（Calendar）を特に重視。各推定に evidence（観測ソース）を1つ以上付す。
- ③ readingBehavior は読書FB（ハイライト/読書ログ/簡易FB）から関心・既読傾向（被り回避）・離脱・評価・許容度・文体嗜好を分析。初回は空でよい。
- 高度なベクトル化は不要。観測テキストから素直に読み取る。前回 profile があれば②③を差分更新する。
出力は ReaderProfile（3層）スキーマのJSONのみ。
```

> **設計規律**: 推定アルゴリズムは作り込まない。企画判断はSTEP2のLLMに委ねる（構想 §6-5「決めるだけ」）。**baseは保持・②③に分析を集中**＝負荷とコストを現状/行動の分析に振る。

## 4. STEP2：企画チーム（★必然性の本丸・絶対削らない）

> **企画構成（3階層の委任・評価構造・2026-06-03 v2）**: 対立構造（本命3体並列＋敵対的選抜）を廃し、**企画リーダー（スコアゲート）→企画担当者（立案）→調査サブ×3（異なる観点の調査）** の3階層に再構成。**設計の起点は「企画書フレーム（§4-2bの8項目）」**で、それを埋めるために必要な調査を逆算したのが調査3観点（A読者局面／B市場競合／Cテーマ知見）。**コンセプト案サブ・タスク分解サブは廃止**（担当者がコンセプト統合まで担う）。担当者が3調査を統合して企画書を作り、リーダーがスコア化して閾値未満なら差し戻す（最高3ラウンド）。**本命（土・水）もセレンディピティ（日）も同一の構造を使い回し**、起動時に `themeKind` を切り替える。必然性の核は「サブが外部実データを取る（Google検索）＋リーダーがスコア閾値で突き返す」（アーキ §7）。

### 4-2a. 企画リーダー＝スコアゲート（×1・最上位・承認者）

**役割**: 企画担当者が上げた企画書を**ルーブリックでスコア化**し、閾値で承認/差し戻しを裁定する。1テーマ（本命 or セレンディピティ）を確定する。
**使用モデル**: Gemini Pro 推奨（最も判断が重い）。
**Firestore**: 承認した `approvedPlan` を `plans/{planId}` に `score`/`rounds`/`themeKind` 付きで保存。
**Langfuse**: **スコア化→閾値未満で差し戻し→再提出のループ（スコアとラウンド数）をトレースに残す**（基準1の生命線・MVPスコープ §5-2）。
**スコア閾値ループの設計（2026-06-03確定）**:
- **スコアルーブリック（4観点・各0〜25点・均等配点・合計100点満点）**：
  ①読者局面への的中度／②既製ビジネス書との差別化／③調査（実トレンド）の反映度／④タイトル・惹きの強さ。4観点の合計が総合スコア。
- **合格閾値＝総合70以上で承認**（**仮置き。運用しながら調整する前提**）。
- **足切りガード**：1観点でも極端に低い（25点満点中10点未満）場合は、総合スコアに関わらず差し戻す。
- **Evalゲートとの共通化**：CIのEvalゲート（基準5・デプロイ時の品質検査）も**この4観点ルーブリックを共通利用**する（モノサシを1つに統一・§8/§9・アーキ§10）。
- **最高3ラウンド**（`max_rounds=3`）：閾値到達で即承認、未到達でも3ラウンド目は最良案を承認し、`score` と「閾値未達フラグ」を記録（無限ループ・コスト暴走の防止／棚を空にしない＝アーキ§3 plans）。
- 「必ず1回却下」のような演出ロジックは入れない。スコア閾値で自然に回るループとして実装する（デモはデモで別途撮る）。

### 入力
- 企画担当者の企画書 `PlanProposal`（当該ラウンド）
- `ReaderProfile`
- `themeKind`（honmei | serendipity・起動時に決定）

### 出力：`LeaderVerdict`
```jsonc
{
  "round": 1,
  "score": 78,                                  // 総合スコア（4観点×各0〜25点の合計・0〜100）
  "scoreBreakdown": {                           // 各観点0〜25点
    "relevance": 22,                            // ①読者局面への的中度
    "differentiation": 18,                      // ②既製ビジネス書との差別化
    "researchUse": 20,                          // ③調査（実トレンド）の反映度
    "titleHook": 18                             // ④タイトル・惹きの強さ
  },
  "belowFloor": false,                          // いずれかの観点が10点未満なら true（足切り＝強制 revise）
  "decision": "approve",                        // approve | revise（score>=70 かつ !belowFloor で approve）
  "rejectionFeedback": null,                    // decision=revise のとき、弱い観点と改善指示を記述
  "approvedPlan": { "planId": "string", "...PlanProposalの全フィールド": "..." }  // approve時のみ
}
```

### プロンプト骨子
```
あなたは企画リーダー（承認者＝スコアゲート）。企画担当者が上げた企画書を、読者プロファイルとの適合で
採点せよ。観点（各0〜25点）＝①読者局面への的中度 ②既製ビジネス書との差別化
③調査（実トレンド）の反映度 ④タイトル・惹きの強さ。4観点の合計を総合スコア（0〜100）とする。
- 総合スコアが閾値（{threshold}＝既定70）以上 かつ どの観点も10点以上なら decision="approve"。
- 総合が閾値未満、または いずれかの観点が10点未満（belowFloor=true）なら decision="revise" とし、
  rejectionFeedback に「どの観点が弱く、どう直すべきか」を具体的に書く。
- 一般論的・ありきたり・独りよがりは強く減点する。読者のリアルな局面に踏み込むほど加点。
- ③は調査サブの実データ（売れ筋・marketGap）が企画に具体的に反映されているかを見る。
出力は LeaderVerdict スキーマのJSONのみ。
```

### 4-2b. 企画担当者＝立案（×1・中間層・責任者）

**役割**: 配下の調査3観点（読者局面・市場競合・テーマ知見）を統合し、**企画書フレーム8項目**を満たす1つの企画書（`PlanProposal`）に練り上げる。コンセプトの統合・複数案の取捨もここで担う（コンセプト案サブは持たない）。リーダーから差し戻されたら弱い観点を最優先で練り直して再提出する（調査サブの再呼び出しは任意・既定はしない）。
**使用モデル**: Gemini Pro（立案は判断が重い。ハイブリッド方針）。

- `rejectionFeedback`（差し戻し時のみ・前ラウンドでリーダーが書いた改善指示）

### プロンプト骨子

### 4-2c. 調査サブ×3（最下層・実行/支援・担当者の配下）

**設計の考え方**: 企画書フレーム8項目（§4-2b）を**埋めるために必要な調査を逆算**した3観点。コンセプト案・タスク分解の生成サブは持たない（コンセプト統合は担当者の責務）。
**使用モデル**: Gemini Flash（B・CはGoogle検索 grounding を有効化）。
**呼び出し**: 初回ラウンドで担当者が起動。差し戻し時の再実行は任意（既定はせず、担当者が既存成果を元に練り直す）。

| サブ | 役割 | 出力（state） | 埋めるフレーム | grounding |
|---|---|---|---|---|
| **A 読者局面リサーチ** | 観測データからこの読者の意思決定・痛点・文脈を深掘り | `subReaderContext` | ②③④ | 不要（内部データ） |
| **B 市場・競合リサーチ** | 最近のビジネス書トレンド・売れ筋・marketGap | `subMarket` | ⑤ | **Google検索**（必然性の核） |
| **C テーマ知見リサーチ** | テーマの実質的論点・フレーム・最新知見 | `subThemeInsight` | ①⑥⑦ | **Google検索** |

### 出力（例：B 市場・競合 `subMarket`）
```jsonc
{
  "themeKind": "honmei",
  "queries": ["新任マネージャー 書籍 2025 売れ筋", "年上部下 マネジメント 本"],  // 実行した検索
  "findings": [
    { "title": "（実在書名）", "point": "この本が扱う論点・売れている理由", "source": "URL" }
  ],
  "marketGap": "売れ筋は一般論中心。『新任×年上の実力者部下』に限定した本は手薄 → 差別化余地"
}
```
> C テーマ知見 `subThemeInsight` は `keyPoints[]`（論点・フレーム・出典）／A 読者局面 `subReaderContext` は `painPoints[]`・`decisions[]`・`evidence[]`（観測ソース付き）を返す。

### プロンプト骨子（B 市場・競合リサーチ）
```
あなたは出版企画の市場調査担当。与えられたテーマ領域について、Google検索で最近のビジネス書トレンド・
売れ筋・関連論点を調べ、企画担当者が差別化を考える材料を返せ。
- 実在の書名・論点を挙げ、可能な限り出典URLを付す（grounding）。
- 「今 何が売れていて、どこに手薄（marketGap）があるか」を必ずまとめる。
出力は subMarket スキーマのJSONのみ。
```
> A（読者局面）/ C（テーマ知見）も同様に、それぞれ「埋めるフレーム」を意識した調査結果を返す専任プロンプトを置く。

> **必然性の最低ライン（5週間版・アーキ §7）**: 「調査サブが外部実データを取得する」「リーダーがスコア化→閾値未満で差し戻し→再提出する」ループがLangfuseに残ればよい。A/B測定（3階層 vs 単一生成）は将来検証。

## 5. STEP3：著者キャスティング

### 5-3a. キャスティング編集者（×1・テーマ確定時に架空著者5人を都度生成＝2026-06-03 v2）

**役割**: 承認企画（1テーマ）に最も合う架空著者ペルソナを、**その場で5人 都度生成**する（固定プールからの選抜ではない）。「このテーマを誰に書いてもらうか」を決め、**著者を“揃える”役**。執筆させるのは次のSTEP4編集長の仕事（本エージェントは書かせない）。
**使用モデル**: Gemini Pro（人格設計は判断が重い。ハイブリッド方針。**1runあたり1コール**で5人を一括生成）。
**必然性補強**: テーマの文脈に完全に合わせた経歴・文体・口癖を持つ著者を生成できる。同一テーマでも読者プロファイル（役職・業界・局面）が変わると生成される著者の切り口・文体が変わる（設計規律「出力が変わる最小構成」をクリア）。
**お気に入り著者の混入**:
- `users/{uid}.favoriteAuthors[]` があれば混入候補として渡す。生成5人のうち何人を混入するかを判断し、混入した著者は `fromFavorite=true` を立てる。
- **MVPは混入比率15%**（生成5人の各枠を約15%の確率で favoriteAuthors から採用＝平均1人弱）。比率・選出ロジック（ランダム性）は今後A/Bで検証。**favoriteAuthors が空なら no-op**（初回ビルドは空＝混入0）。
**実装メモ**:
- 出力スキーマ検証必須（員数5人を守らせ、不一致なら再試行）。ADK構造化出力で吸収。
- 生成ペルソナは `personas/{personaId}`（`ephemeral=true`）に保存。`book.authorPersonaId` が参照する正本になる（棚の有効期間中は保持）。
- 既存 `著者ペルソナ集.md` の15件は固定プールとしては使わない。生成インスピレーション素材として要約をプロンプトに添付してよい（任意・必須ではない）。

### 入力
- `approvedPlan`（1テーマ・`recommendedAuthorTypes`・`themeKind` を含む）
- `ReaderProfile`（文体嗜好 `readingBehavior.stylePreference` を生成ペルソナの主軸著者の文体設計に反映）
- `users/{uid}.favoriteAuthors[]`（任意・混入候補。MVPは約15%の確率で採用／空なら no-op）
- `著者ペルソナ集.md` 要約（任意・生成インスピレーション素材。コピーはしない）

### 出力：`GeneratedPersonaSet`
```jsonc
{
  "planId": "承認企画のID",
  "themeKind": "honmei",                         // honmei | serendipity
  "personas": [
    {
      "personaId": "string",                     // Firestore自動IDでよい
      "name": "架空の著者名",
      "voiceStyle": "文体軸＝語り口・思考スタイル（ロジカル/思想的・哲学的/感覚的・情緒的/泥臭い・現場/学術的 等）",  // ★分割
      "format": "文章カテゴリ・形式（ストレートな自己啓発書/小説・物語形式/エッセイ形式/対話・問答形式 等）",       // ★新規
      "persona": "経歴・口癖・思想・原体験（すべて架空・★他項目より重厚に・具体的に・生々しくリッチに作り込む）",  // ★リッチ化

      "pastBooks": [],                           // 初回は空。ファン化でランタイム蓄積

      "ephemeral": true                          // 棚有効期間中のみ保持フラグ
    }

  ],
  "reason": "5人を voiceStyle（文体軸）× format（文章形式）の2軸で分散（例：ロジカル×自己啓発／思想的×エッセイ／感覚的×小説形式…）。読者の stylePreference に主軸1人を寄せた"
```

### プロンプト骨子
```
あなたは著者設計（キャスティング編集者）の専任エージェント。承認企画のテーマ・コアメッセージ・読者プロファイルに
最も合致する架空の著者ペルソナを5人 生成せよ。

- 員数5人を厳守し、reason を付す。
出力は GeneratedPersonaSet スキーマのJSONのみ。
```

## 5-2. STEP4：編集ループ（プレビュー編集・編集長 ⇄ 著者5人・1ラウンド）★新設

```
①title ②subtitle ③deliveryReason（今あなたは＝なぜこの本を届けるか・観測ソースに触れる）
④problemToSolve（解決する課題）⑤coreMessage（核心メッセージ）⑥agenda（目次＋各章一行サマリー）⑦prefaceSample（はじめに）
- あなたの voiceStyle・format・思想が前面に出る独自の切り口にすること（ファン化の見せ場）。
- ③④では読者の具体的状況（{ReaderProfile.currentWork.currentSituation / challenges}）に名指しで踏み込む（解像度の証拠）。
- title の人数等は読者の base.orgScale に整合させる。タイトルは万人受け不要・エッジを効かせる。
- editorFeedback があれば、指摘点を最優先で直す。
出力は BookDraft スキーマのJSON（status="draft"）のみ。本文はまだ書かない。
```

### 5-2b. 編集長＝プレビュー採点・差し戻し（×1・STEP3とは別エージェント）
**役割**: 各著者のプレビューを**プレビュー3観点**で採点し、合格（緩め）ならそのまま、不足なら `editorFeedback` を付けて1度だけ差し戻す。
**使用モデル**: Gemini Pro。

#### プレビュー3観点（各0〜25点・**合計75**／合格は緩めの閾値）
| 観点 | 見るポイント |
|---|---|
| ① 生の情報・読者状況の反映 | 一般論でなく、読者分析（この人の局面）を捉えているか |
| ② 著者ペルソナの特徴が前面 | 設定した著者の個性・思想が核心メッセージ・アジェンダに強く出ているか |
| ③ キャッチー・端的・タイトルの惹き | 全体が目を引くワーディングで、タイトルが面白いか |

> STEP2企画4観点とは別軸。特に**②著者ペルソナの前面化**がSTEP4固有。**合格閾値＝総合50/75（仮置き・I-18／企画リーダー 70/100＝0.70 より緩い 0.67）＋足切り：いずれかの観点が10点未満なら revise**。明らかな不足のみ1Rで弾く＝棚を空にしない。本文の厳しい作り込みはモードB（§7）で行う。

#### 出力：`EditorVerdict`
```jsonc
{
  "bookId": "string",
  "round": 1,
  "score": 70,                                 // 4観点合計（0〜75）。例は 24+23+23
  "scoreBreakdown": { "rawInsight": 24, "personaForward": 23, "catchiness": 23 },
  "decision": "approve",                       // approve | revise（総合<50/75 or いずれか<10 で revise・最高1R）
  "editorFeedback": null                       // revise時のみ：弱い観点と直し方
}
```

> **重要な分離（構想 §6-2b）**: 作家＝**人格データ**（動かない）、執筆エンジン＝**1種類**（実行時に人格を着替える）。作家ごとにエージェントを常駐させない。編集長は1体で5著者を順に見る。

## 6. STEP5：装丁（軽エージェント＋Imagen）

**役割**: ビジュアル方針を判断し、Imagenで表紙を先行生成（棚を賑やかにする）。
**使用モデル**: Flash（方針判断）＋ Imagen on Vertex AI（生成）。
**Firestore/GCS**: 画像をCloud Storageに保存し、`books/{bookId}.coverUrl` を更新。

### 入力
- `BookDraft`（`title`・`coreMessage`）＋ `persona`（`voiceStyle`・`format`）

### 出力
```jsonc
{
  "bookId": "string",
  "coverPrompt": "Imagenへ渡す英語プロンプト（書影・ビジネス書風・トーン指定）",
```

### プロンプト骨子（方針判断→Imagenプロンプト生成）
```
あなたは装丁担当。本のタイトル・核心メッセージ・著者の voiceStyle/format から、ビジネス書として
```

## 7. モードB：本文編集ループ（編集長 ⇄ 著者・最高3ラウンド）

**役割**: ユーザーが予約した本だけ、**編集長⇄著者の編集ループ**で本文（約100ページ）を作り込む。STEP4プレビュー編集の本文版＝より重い最高3R。
**トリガー**: `books/{bookId}.status: draft→reserved`（ユーザー選択）→ Pub/Sub。
**★予約上限**: 1ユーザーあたり**同時最大5冊**（MVP・デモは1冊／単位＝reserved+writing の合計＝I-16解決）。本文Pro生成のコストを天井留め（API契約仕様 §3で `POST /reserve` がチェック）。
**使用モデル**: **著者（章生成）・編集長（採点）ともに Gemini Pro**。予約制＋同時上限5冊により本文をフル執筆するのは限られた冊数だけなので、最高品質を集中投下できる。本文は読者が読む唯一の成果物＝品質が直接体験価値（基準4）。
**Firestore/GCS**: 本文MDをCloud Storageへ、`books/{bookId}.bodyUrl` 更新、`status: writing→published`。

### 入力
- `BookDraft`（`agenda`・`coreMessage`）／`persona`・`approvedPlan`・`ReaderProfile`

### 処理フロー（編集ループ・最高3R）
```
1. 著者：アウトライン確定（agendaを節レベルに展開）→ 章ごとに本文生成（人格を着る・前章要約で一貫性維持）
2. 編集長：**本文ルーブリック5観点（I-12確定）**で採点。弱い章を特定し editorFeedback を返す
3. 著者：指摘された【弱い章のみ】改稿（全文再生成しない＝コスト抑制）
4. 2〜3を最高3ラウンド（閾値到達 or 3R到達で確定）
5. 通し校正（Pro）→ 本文MDをGCS保存 → status: writing → published
```

### 出力
```jsonc
{
  "bookId": "string",
  "bodyUrl": "gs://.../books/{bookId}.md",
  "status": "published",
  "pageEstimate": 100,
  "editRounds": 2                              // 編集長との改稿ラウンド数（Langfuseに残す＝必然性の証跡）
}
```

### 本文ルーブリック5観点（編集長の採点軸・I-12確定／プレビュー3観点とは別・**合格＝総合70/100 仮置き**・運用調整）
| # | 観点 | 見るポイント |
|---|---|---|
| ① | 構成の一貫性 | 章間の論理の流れ・前章との接続が破綻していないか |
| ② | 各章の掴み（引き込み） | 冒頭・見出しで引き込み、離脱させないか |
| ③ | 読者状況への的中 | 読者の局面（currentWork）に踏み込み、"的中の一節"があるか |
| ④ | 著者ペルソナの一貫性 | voiceStyle / format がブレず前面に出ているか |
| ⑤ | 実践性・具体性 | 具体例があり**行動に落とし込める**か（一般論・水増しでない） |

> **執筆方針**: 量は約100ページ／品質は要所に傾斜配分（タイトル・入荷理由・序文・冒頭章・"的中の一節"を担保、本文の大半は量を確保）。水増し禁止。**改稿は弱い章のみ**（全文再生成はコスト的にしない）。粒度選択（フル/要約）はStretch。

## 8. Eval（CI品質ゲート・L4）

**役割**: 企画を LLM-as-judge で採点し、CIのデプロイゲートにする（MVPスコープ §9）。**採点は企画リーダーと同じ4観点ルーブリックを共通利用**（§4-2a・モノサシを1つに統一・2026-06-03確定）。
**使用モデル**: Gemini（judge）。
**走る場所**: GitHub Actions（プロンプト/ペルソナ/企画ロジックのpush時）。

### 入力：Eval Set（8件）
```jsonc
{
  "id": "eval_01",
  "readerProfile": { /* ReaderProfile（サンプルDrive由来） */ },
  "plan": { /* PlanProposal */ },
  "expectedBand": "high",        // high(≥70) | low(≤40) | serendipity(30〜60)
  "kind": "本命"
}
```
内訳：高関連4／ずれ2／セレンディピティ2（MVPスコープ §9-3）。

### judge出力（4観点ルーブリック・各0〜25点／合計0〜100）
```jsonc
{
  "id": "eval_01",
  "score": 78,                                  // 総合（4観点合計・§4-2a と同一）
  "scoreBreakdown": { "relevance": 22, "differentiation": 18, "researchUse": 20, "titleHook": 18 },
  "reason": "役職・組織規模・局面に踏み込み一般論でない。LangfuseにreasonをLog"
}
```

### ゲート判定
- 本命企画：**総合 < 70 でデプロイ停止**（リーダー閾値70と同一基準）。1観点でも10点未満なら警告。
- セレンディピティ：中レンジ（総合30〜60目安）のレンジ外なら警告（停止しない）。
- **8件中7件パス（87.5%）で通過**。

### judgeプロンプト骨子（4観点・§4-2a 共通ルーブリック）
```
読者プロファイルと生成企画を読み、4観点（各0〜25点）で採点せよ。
①読者局面への的中度 ②既製ビジネス書との差別化 ③調査（実トレンド）の反映度 ④タイトル・惹きの強さ。
4観点の合計を総合スコア（0〜100）とする。
- 一般論的（最大公約数的）なら減点。読者のリアルな状況に踏み込むほど加点（＝課題§2-1①と直結）。
出力は { "score": 0-100, "scoreBreakdown": {...}, "reason": "..." } のJSONのみ。
```

## 11. フロント⇔バック連携方式（§10-5・MTG 2026-06-05で確定＝G1-3）

> 原則：**Firestoreを正本（source of truth）とし、フロントは可能な限り直接読み書き**。バックエンド処理を伴う/機密な操作だけ薄いCloud Run APIを通す。スタックがCloud Run中心のため、FirestoreトリガーのCloud Functionsは足さない方針。

| 操作 | 方式 | 担当 |
|---|---|---|
| 棚（draft一覧）・本詳細・アジェンダの読み取り | **Firestoreリアルタイム購読**（API不要） | 鉄田（フロント） |
| ハイライト・簡易FB・★保存 | **Firestore直書き**（セキュリティルールで保護） | 鉄田（フロント） |
| 予約（`draft→reserved` ＋ Pub/Sub執筆発火） | **POST /reserve**（Cloud Run）：Firestore更新＋Pub/Sub発行を1エンドポイントで | 友人 |
| OAuth連携（Drive/Calendar/Tasks のトークン交換） | **Cloud Run API（バックエンド必須）** | 友人 |
| 手動トリガー（デモ用に企画バッチを起動） | **Cloud Run API**（Cloud Run Jobを叩く） | 友人 |

> **役割分担との整合（アーキ §5）**: フロント（鉄田）はFirestore SDKで大半を完結、バックは最小API3本で済む＝工数を物理分離できる。
> **✅ MTG 2026-06-05確定（G1-3）**: 予約のPub/Sub発火は **POST /reserve 方式（明示API）** で行う（Firestoreトリガーは不採用＝スタックにCloud Functionsを足さない）。
