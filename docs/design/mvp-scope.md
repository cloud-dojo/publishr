## 1. MVPの一文定義（North Star）

> **登録時に初期プロフィール（業界・職種・役職・最近の関心）を入力し、Drive・カレンダー・Googleタスクを観測して、週3回 自律的に「本命企画×5冊（土・水）＋ セレンディピティ企画×5冊（日）＝ 週15冊」が棚に入荷し、入荷理由が付き、選ぶと1冊が時間をかけて本格執筆されて読め、ハイライト・簡易FB・お気に入り著者の保存ができ、EvalゲートつきCIで自動デプロイされる——を「デモ動画で見せられる」状態。**

判断基準はただ一つ：**W5のデモ動画の1カットに映るか**。映らない機能はMVPに入れない。

キラーフレーズ：**百万部のベストセラーより、あなたのための一冊。**

## 2. なぜこのMVPか（課題 → ターゲット → コア価値）

> 審査基準2は「課題→背景→対象ユーザー→価値」の一貫性・新規性を独立評価する。スコープの全機能はここに紐づく。

### 2-1. 中核課題（2層）

**① 構造的課題（最重要）：出版ビジネスは「最大公約数」しか出せない。**
本（特にビジネス書）は商業的に成立させるため、最大多数に売れる**普遍的・一般論的な内容に最適化**される。だが現実の読者の状況は千差万別——同じ「マネジメント」でも、30人を率いるマネージャーと、部下2〜3人の新任マネージャーでは必要なノウハウは全く違う。業界・業種・会社のフェーズによっても事情は変わる。**「最大公約数を取りにいく出版モデル」と「個人のリアルな状況に合致するシャープなノウハウ」は、ビジネスモデルの構造としてそもそもミスマッチ**している。だから自分の置かれた環境にぴったり的中する一冊には、構造的に巡り会えない。**Publishrはこの構造そのものに着目し、改善する。**

**② 認知的課題：自分の関心は自分では言語化できない。**
既存の本生成サービスはすべて「ユーザーが指示する」前提で、指示できる人にしか価値が届かず、言語化できない関心を取りこぼす。

→ Publishrは、**観測で①の「あなただけのリアルな状況」を捉え、自律企画で②の「言語化の壁」を越える。**

### 2-2. ターゲット（確定：ビジネスパーソン）
情報感度は高いが、能動的に本を探す時間・気力がない人。自分の関心・仕事状況を汲み取ってほしい人。「自分だけの」所有感に価値を感じる人。
- 戦略的理由：ビジネス書は「型」（課題→原因→FW→事例→アクション）があり、AIが最低コストで"それらしい一冊"を成立させやすい＝執筆品質リスクを構造的に低減できる勝ち筋。

### 2-3. コア価値（課題と対の形）
| 価値 | 課題との対応 |
|---|---|
| **あなた専用の最適化**（最重要） | 最大公約数(課題①)の逆。役職・組織規模・業界・局面まで踏まえ、あなたのリアルな状況に的中させる＝市販本が構造的に出せない一冊 |
| **言語化不要** | 指示しなくても観測データから関心を推定して企画（課題②） |
| **編集された深さ** | 断片でなく、構造化された50〜100ページの一冊として届く |
| **関係性** | 仮想著者をファン化し、続編・同テイストで関係が育つ |
| **多様性** | 同一テーマでも複数の著者が異なる切り口で書く |
| **偶然性（セレンディピティ）** | 本命テーマに加え「興味があるか微妙だが刺さるかも」な教養テーマ（宗教・IT・哲学等のリベラルアーツ）をあえて出題。フィルターバブルを破る（8:2原則） |
| **待つ体験** | 予約後ゆっくり執筆。"届くのを待つ"こと自体が所有感・期待を育てる |

### 2-4. 新規性（一行）
**ユーザーが指示しないのにエージェント側が「いま何を読ませるべきか」を自律判断して出版する（プッシュ型・自律企画）。** 同一構想は競合調査で未発見。新規性は「自律企画 × 仮想著者 × 観測」の組み合わせ方にある。

## 3. スコープIN（＝デモに映る最小セット）＋ Definition of Done

> DoD＝「この線を越えたら完成」。"映る"だけでなく"完成"の判定基準。
> **全体の流れ**: 曜日別トリガー（土/水=本命、日=セレンディピティ）→ **3階層の企画チーム**（調査3観点 → 企画担当者が企画書8項目に立案 → 企画リーダーがスコア化し閾値未満なら差し戻し・最高3R）→ 1テーマ確定 → **著者キャスティング5人 都度生成** → **編集長⇄各著者の編集ループ（1R）で「はじめに＋核心メッセージ＋アジェンダ」を作り込み** → **1run5冊が棚に入荷** → ユーザーが選ぶ → 予約（同時最大5冊）→ **編集長⇄著者の本文編集ループ（最高3R）で本格執筆** → 読む → ハイライト／FB／お気に入り著者保存。
> **企画チームの構成（3階層・2026-06-03 v2）**: 対立構造を廃し、企画リーダー（スコアゲート・Pro）→企画担当者（立案・Pro）→調査サブ×3（A読者局面／B市場競合=Google検索／Cテーマ知見=Google検索・Flash）。企画書フレーム8項目を先に固定し調査を逆算（コンセプト案・タスク分解サブは廃止）。本命/セレンディピティは同一構造を使い回す。詳細なI/O契約は `エージェントIO契約.md` §4。著者キャスティング(STEP3)・編集ループ(STEP4/モードB)は同 §5-§7。

## 4-b. 💭 攻めのスコープ候補（MVP外・採否は今後判断）

> 「入れたら尖るが、5週間に収まるか未確定」のメモ。スコープインの判断は別途。

| 候補 | 内容 | 検討メモ |
|---|---|---|
| **オンデマンド企画** | ユーザーが「このテーマで」と指定したら、定期トリガーと無関係にその場で企画→入荷 | プッシュ型(自律)に対する"プル型"。攻めの差別化になるが、自律性の主役を食わないか要検討 |
| **読者分析の時系列可視化** | 読者プロファイルの変遷を時系列で閲覧（思考の変化が追える） | 体験価値・新規性として面白い。Firestoreに履歴を持てば実現可。UI工数とトレードオフ |
| **読むときの粒度選択**（フル／要約） | 執筆は本格(長文)で行い、**読む段階**でフル版/要約版を選べる | 「執筆時の粒度選択」(OUT)とは別物。読書UI側の派生表示。Stretch候補 |
| **フラッシュモード** | 「すぐ読みたい」ユーザー向けに高速・短文執筆を発注時に選べる | 予約=待つ体験の例外オプション。今回は本格執筆前提とし、選択肢として置くだけ |

## 5. 受け入れ基準（Definition of Done — プロジェクト全体）

### 5-1. デモで見せる"黄金の1カット"
```
【初回利用時】
  アプリを開く → 初期プロフィール入力（業界/職種/役職/最近の関心・タップのみ・30秒）
  → Drive/Calendar/Tasks のOAuth同意 → 完了

【2回目以降（黄金の1カット）】
ある朝、アプリを開く
  → 棚に、昨日なかった新刊が"入荷"（自律トリガーの成果）
     ・本命（土・水）：「ロジカルシンキング」5冊（同テーマを5人の著者が別切り口で）
     ・セレンディピティ（日）：「宗教入門」「IT再入門」など教養5冊
  → 各表紙に入荷理由（本命「最近マネジメントに悩んでいるようなので…」／
     偶然性「論理思考が好きなあなたは、実は哲学も刺さるかも」）
  → 1冊を選ぶ → 「予約しました（執筆中）」
  →（動画は画面を切り替え）開くと「入荷しました」＝50〜100pの本文が読める
  → 気に入った一節をハイライト → 「★これは役立った」をポチッ → 気に入った著者を「保存」（ワンクリック）
  →（裏でLangfuseに スコア差し戻し判断・調査サブの検索が記録され、CIのEvalゲートが回る画面を補足カット）
```
> **デモは自作の編集動画**。執筆に実時間がかかっても、カット編集で「予約→入荷」を表現すればよい。**「数十秒で完成」を見せる必要はない**（時間軸の制約は緩い）。

## 9. Eval設計（品質ゲート詳細）

> 基準5・Observability L4（継続評価）の核。**企画リーダーと同じ4観点ルーブリックを共通利用**し（2026-06-03確定）、LLM-as-judgeで採点する。

### 9-1. 評価対象（何を測るか）
**企画スコア（4観点・各0〜25点／合計0〜100）** ＝ 生成された企画が読者プロファイルにどれだけ的中しているか。観点＝①読者局面への的中度／②既製ビジネス書との差別化／③調査（実トレンド）の反映度／④タイトル・惹きの強さ。
- **主対象：本命企画** … 高スコアであるべき
- **副対象：セレンディピティ企画** … "近すぎ/遠すぎ"を弾く（中レンジが適正）

### 9-2. 採点方法（LLM-as-judge）
| 項目 | 内容 |
|---|---|
| judge | Vertex AI Gemini |
| 入力 | 読者プロファイルJSON ＋ 生成企画 |
| ルーブリック | §4-2a の4観点（各0〜25点）。**一般論的（最大公約数的）なら減点、リアルな状況に踏み込むほど加点**（＝課題§2-1①と直結） |
| 出力 | `{ score: 0–100, scoreBreakdown: {...}, reason: "..." }`（reasonをLangfuseに記録） |

### 9-3. Eval Set（8件）
サンプルDriveのプロファイルに対する (プロファイル, 企画, 期待スコア帯) のペア。

| 種別 | 件数 | 期待スコア | 狙い |
|---|---|---|---|
| 高関連であるべき（良い企画） | 4 | ≥ 70 | 良企画を正しく通す |
| 低関連であるべき（ずれた企画） | 2 | ≤ 40 | **ゲートが"ちゃんと落とす"証明** |
| セレンディピティ適正（教養企画） | 2 | 30〜60 | 近すぎ/遠すぎを検出 |

> 例（プロファイル「食品メーカーの新任マーケ課長／部下7名・年上の佐藤さん／6/5役員報告を控え自信不足」＝デモのペルソナ佐倉美咲。詳細は `demo/Eval品質ゲート.md`・`demo/fixtures/eval_set.json`）：
> ・高関連の良企画＝「年上のベテラン部下に、どう任せ・どう評価するか」
> ・低関連のずれ企画＝「新入社員のためのビジネスマナー入門」→ 落ちるべき
> ・セレンディピティ適正＝「リーダーのための東洋哲学入門」→ 中レンジ

### 9-4. しきい値（ゲート判定）
- **本命企画：総合 < 70 ならデプロイ停止**（リーダー閾値70と同一基準。1観点でも10点未満なら警告）
- セレンディピティ企画：30〜60 のレンジ外なら警告（停止はしない）
- 判定単位：Eval Set **8件中7件パスで通過**（87.5%）

### 9-5. CI挟み込み（どこで走るか）
GitHub Actions で、**プロンプト／作家ペルソナ／企画ロジックの変更をpush**したときにEval Setを実行 → ゲート判定 → passならCloud Buildへ。failならデプロイ停止＋結果をLangfuseに記録。

> ⚠️ 友人MTG議題：LLM-as-judgeのCI組み込み（GitHub Actionsでのスコア取得）が5週間で実装可能か。
