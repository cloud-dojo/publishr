# Publishr 技術アーキテクチャ（ハッカソン現実版）

> 📑 全体の目次・真実源マップは [正本マップ](../README.md)／未確定論点は [open-issues.md](../planning/open-issues.md)。

> **位置づけ**: 「あなた専属の、AI出版社。」Publishr を、ハッカソンで実際に動かす前提で整理した技術設計・フィージビリティ評価・週次ロードマップ。
> **開発期間（重要）**: 6/1〜7/10。大会期間は約2.5ヶ月だが、**鉄田の実働可能期間は約5週間（1ヶ月＋1週間）**。期間2.5ヶ月と実働5週間を混同しない。
> **体制**: エンジニア友人1名＋鉄田（Claude Codeでコーディング可）＝**実働約1.5〜2名**。
> **方針**: 仰々しく作らない。審査の核（自律企画が動く・書店UIで世界観が伝わる・デモが録れる）に一点集中し、それ以外は大胆に削る。
> **確定事項**: エージェント基盤=**ADK主軸**／観測=**Langfuseマネージド（Cloud）**／企画=**3階層の委任・評価構造（企画リーダー＝スコアゲート／企画担当者／サブ実行者）**。
> **最終更新**: 2026年6月3日

## 0. 結論（先に）

- **5週間でMVPは到達可能。ただし「全部入り」は捨てる前提。** 機能の引き算が設計の本質。
- 「自律性」の技術的正体は派手な機構ではなく **Cloud Scheduler の定期トリガー**。基準1で見せたい自律性は枯れた仕組みで実現でき、ここが最大の安心材料。
- 到達目標（MVP）: Driveを観測して**自律的に企画が棚に並び**、入荷理由が付き、選ぶと1冊が執筆されて読め、FBでき、EvalゲートつきCIで自動デプロイされる——を**デモ動画で見せられる**状態。
- 諦める前提: 5ステップ・フルチーム／ES必然性／改稿ループ／粒度選択／学習の実データ多サイクル → 将来構想・ピッチ言及に回す。
- **マルチユーザーを前提インフラとして設計する**: Firebase Auth＋Firestore のデータモデルは複数ユーザー（`ownerUid` による所有権分離）を最初から組み込む。デモは1ユーザーで動かすが、スケールアウトの障壁を最初から作らない（本番スケール・課金は将来）。
- 最大の勝負どころ: **W2で薄くてもE2Eを1本通せるか。**

## 1. 全体構成：2つの実行モードに分離する

性質の異なる2処理が同居している。混ぜると破綻するので分離が設計の出発点。

```
─────────────────────────────────────────────────────────
 モードA：自律企画（プッシュ型・定期バッチ）← 自律性の心臓部
─────────────────────────────────────────────────────────
 Cloud Scheduler（週3回・曜日別ジョブ ※曜日はW3実測後に微調整）
   ├ 土 朝一：STEP0観測 → STEP1読者分析（週1回のみ）→ STEP2本命企画 → STEP3-4
   ├ 水    ：STEP2本命企画（土朝のprofile再利用）→ STEP3-4
   └ 日    ：STEP2セレンディピティ企画（本命と同一の3階層構造／土朝profile再利用）→ STEP3-4
      │ 起動
      ▼
 Cloud Run Job「企画会議」（ADKマルチエージェント）
   STEP0 観測（バッチ・非エージェント／週1回・土朝のみ）
     ┌─ 確定インプット（MVP3ソース＝Drive＋Calendar＋Tasks） ──────┐
     │ Drive（drive.file）: 業務資料・関心フォルダのテキスト抽出      │
     │   └ 選択は Google Picker 経由・**フォルダ単位**でユーザー指定   │
     │     （drive.fileは走査不可／G1-13＝MTG 2026-06-05で確定）       │
     │ Google Calendar（calendar.readonly）: 直近予定＝業務局面シグナル│
     │ Google Tasks（tasks.readonly）: 未完了/完了タスク＝関心・課題   │
     │ 読書ログ: 読了率・離脱・滞在時間（暗黙的FB）                    │
     │ ハイライト: 読書中の選択箇所（最高純度の関心シグナル）          │
     │ 簡易FB: 選択肢ポチポチ式の明示的フィードバック                  │
     │ initialProfile: ユーザー登録時の初期プロフィール（選択式）      │
     │   └ 初回企画生成時のみ主要シグナルとして活用                    │
     └──────────────────────────────────────────────────┘
     ※Calendar/Tasksの取得内容→STEP1読者分析への反映仕様の詳細は AGENT契約 STEP0/STEP1（G1-14）
     ┌─ 拡張インプット候補（MVP対象外・後工程で絞り込み）─────────┐
     │【業務局面】Gmail件名 / Slackチャンネル名                      │
     │【思考の現在地】Notion / Google Keep / Kindleハイライト / Pocket│
     │【知識ギャップ】ChatGPT・Claude履歴 / YouTube履歴 / Podcast    │
     │             Webブラウザ履歴（最強・取得難・拡張必要）          │
     └──────────────────────────────────────────────────┘
     └─ すべて決定論的処理＝ツール。判断しないのでエージェントにしない
   STEP1 読者分析（1エージェント・**Pro**／週1回・土朝のみ）
     └ ReaderProfile（**3層**）確定：base（業界/職種/役職/orgScale/読書傾向＝保持）／currentWork（局面/課題/控える局面＝週次分析の主戦場）／readingBehavior（既読傾向/ハイライト/評価＝学習シグナル）。base以外を1週間有効で更新
   STEP2 企画チーム ★必然性の本丸（削らない）＝3階層の委任・評価構造
     ├ Layer1 企画リーダー（Pro）：担当者の企画書を4観点スコア化。閾値未満なら差し戻し（最高3ラウンド）
     ├ Layer2 企画担当者（Pro）：調査3観点を統合し企画書（8項目フレーム）を立案。差し戻しは弱い観点を練り直す
     └ Layer3 調査サブ×3（Flash）：A読者局面／B市場・競合（Google検索）／Cテーマ知見（Google検索）
        ※企画書フレームを先に固定し、それを埋める調査を逆算した3観点（コンセプト案・タスク分解サブは廃止）
        ※本命（土・水）もセレンディピティ（日）も同一の3階層構造を使い回す
   STEP3 著者キャスティング（キャスティング編集者・Pro）：テーマに合う架空ペルソナを5人 都度生成
        （favoriteAuthorsは起用候補・配本ごと約25%で1冊／著者を“揃える”役）
   STEP4 編集ループ（編集長 ⇄ 著者5人・1ラウンド）★新設：
        編集長が各著者に「はじめに＋核心メッセージ＋アジェンダ」を書かせ、プレビュー3観点で採点→1R改稿
        （ユーザーが選ぶ画面の質を担保・著者=Pro/編集長=Pro）
   STEP5 装丁：Imagenで表紙生成（先行生成で棚を賑やかに）
      │
      ▼
 Firestoreに「企画状態(draft)」で書き込み（本文は未執筆）
      │
      ▼
 ユーザーが翌朝アプリを開くと「入荷」している（＝自律性の体験）

─────────────────────────────────────────────────────────
 モードB：後追い執筆（オンデマンド・予約トリガー）
─────────────────────────────────────────────────────────
 ユーザーが「著者版」を選ぶ＝予約（＝最強の学習シグナル・★予約上限 同時最大5冊）
      │ Firestoreの状態を reserved に更新 → Pub/Sub発火
      ▼
 Cloud Run（本文編集ループ：編集長 ⇄ 著者・最高3ラウンド）
   ├ 著者エンジン（人格データを"着て"本文生成：アウトライン→章→節）
   ├ 編集長：本文ルーブリックで採点 → 弱い章のみ改稿（最高3R・全文再生成しない）→ 通し校正
   └ 本文をCloud Storage、メタをFirestoreに「完成(published)」で保存
      │
      ▼
 フロントが状態を購読 → 「執筆中→入荷しました」に変わる
```

> 構想6-2dで確定した分岐: ①選抜ゲートは企画リーダー1段（編集長を独立二段目にしない）。②STEP3の著者版選抜は**ユーザーが選ぶ**（＝予約の瞬間・学習シグナル・ファン化を兼ねる）。③アジェンダは**著者が書く**（文体の個性＝ファン化の見せ場）。

> **なぜ予約制がアーキテクチャを救うか**: 全書籍フル執筆はコスト・時間が爆発する。予約制なら定期バッチで生成するのは「企画＋表紙＋序文サンプル」という軽い成果物だけ。重い本文生成はユーザーが選んだ1冊にだけ走る。**期間が短いほど効く設計**（棚は事前生成サンプルで賑やかに見せ、本文を真面目に書くのはデモで開く1冊だけでよい）。

## 2. GCPサービス・マッピング（何を・なぜ）

| レイヤ | 採用サービス | 役割 | なぜこれ／代替 |
|---|---|---|---|
| フロント | Next.js or Firebase Hosting | 書店UI（デモの主役） | **鉄田がClaude Codeで実装**。テンプレ/UIライブラリ前提で最短化 |
| API/BFF | Cloud Run（Python/FastAPI） | フロントとエージェントの橋渡し | サーバーレス。**GCEは不要（過剰）** |
| 企画バッチ | Cloud Run Job + Cloud Scheduler | モードAの自律トリガー | 自律性の心臓部。cron相当で十分 |
| 執筆ワーカー | Cloud Run + Pub/Sub | モードBの非同期執筆 | 予約→発火→裏で執筆。タイムアウト回避 |
| エージェント | **ADK**（必要時のみLangGraph部分併用） | 企画会議・執筆 | ハッカソン必須技術。基準1の主役。Google深活用 |
| LLM | Vertex AI Gemini（ハイブリッド：判断が重い工程はPro／観測寄りはFlash） | 推論全般 | 即時性不要なので品質重視でProを基本、抽出寄り（STEP1・調査サブ・装丁）はFlashで圧縮。詳細は `エージェントIO契約.md` §9 |
| 画像 | Imagen on Vertex AI | 表紙生成 | 装丁。企画段階で先行生成 |
| 状態・メタ | Firestore | 企画/予約/執筆/完成の状態管理・FB | リアルタイム購読が書店UIと好相性 |
| 本文・画像 | Cloud Storage | 本文(MD/JSON)・表紙画像 | 大きいテキスト/バイナリはDBに入れない |
| 検索 | Elasticsearch（Elastic Cloud） | 続編記憶・既出概念の横断検索 | **今回は外す判断もあり**（§9参照） |
| 観測 | **Langfuse Cloud（マネージド）** | トレース・Eval Set | GKE自前運用は不採用（§8） |
| 評価 | **Vertex AI Gen AI Evaluation Service**（GEAP/Vertex AI） | Eval品質ゲート（LLM-as-judge・4観点ルーブリック・8件） | 自作judgeでなくマネージド評価サービス。CIの品質ゲートに使用（§10）。詳細＝`publishr_other/GEAP②_EvalService具体化.md` |
| 認証 | Firebase Auth + Google OAuth | ログイン＋各種スコープ同意 | Drive→Calendar→Gmail の順にスコープ追加が自然な拡張路線 |
| 外部連携（拡張） | Google Calendar API / Gmail API / Notion API / Kindle / Pocket / YouTube Data API 等 | 拡張インプット取得 | 採否は後工程で決定。Calendar が最優先候補（同じOAuthで追加できる） |
| CI/CD | GitHub Actions → Cloud Build → Cloud Run | 自動デプロイ | 基準5・DevOps証明 |
| IaC | Terraform（コア資源のみ） | インフラ定義 | 時間が無ければ一部gcloud手動も許容 |

> **GCE（Compute Engine）は使わない。** Cloud Runで完結。「仰々しくしない」方針に合致し、審査員（佐藤氏）には適切な技術選定と映る。

## 3. データモデル（Firestoreの状態機械）

> 📌 **データモデル（Firestoreスキーマ）の真実源（Source of Truth）は本節。** 他docはここを参照し、フィールド定義を重複させない（エージェントIO契約 §付録＝出力→保存先の対応のみ／Firestoreセキュリティルール §2＝ルール固有の論点のみ）。

プロダクトの本質は「本の状態が時間で遷移する」こと。

```
plans/{planId}        ← STEP2が産んだ承認企画（コンセプト）
  ├─ ownerUid          ← 所有者（Firebase Auth uid）＝セキュリティルールの所有権判定（Firestoreセキュリティルール §2）
  ├─ themeKind: honmei | serendipity   ← 本命（土・水）かセレンディピティ（日）か
  ├─ reason: "最近○○に関心が高まっているため"  ← 入荷理由＝解像度の証拠(F3)
  ├─ coreMessage, readerSituation, agendaOutline[]
  ├─ score, rounds     ← 企画リーダーの最終スコアと、閾値到達までに要したラウンド数（最高3）
  └─ recommendedAuthorTypes[]

books/{bookId}        ← STEP3が産んだ「著者版」（同一企画×著者ごとに別レコード）
  ├─ ownerUid          ← 所有者（Firebase Auth uid）＝所有権判定＋予約API(API契約仕様 §3)
  ├─ planId（どの企画から派生したか）
  ├─ status:  draft → reserved → writing → published   ← 状態機械
  ├─ authorPersonaId, title, prefaceSample, agenda[], coverUrl(GCS)
  ├─ bodyUrl(GCS), granularity: full|summary|excerpt〔Stretch〕
  ├─ feedback: { rating, wantsSequel, readPercent, dropped }  ← 読書ログは feedback に集約（I-9＝MTG 2026-06-05確定。dwellSecはMVP対象外。フロント直書きは onlyChanged(['feedback']))
  └─ (sub) highlights/{highlightId}  ← ハイライトのみサブコレクション（I-9）

users/{uid}/observations/{YYYY-MM-DD}   ← STEP0観測が吐く ObservationBundle（I-19＝MTG 2026-06-05確定）
  ├─ collectedAt        ← 観測実行時刻（docID＝土朝runの日付で冪等・再runは上書き）
  ├─ drive[]            ← Drive抜粋（1ファイル約4,000字×約10件・生テキストをインライン保存）
  ├─ calendar[]         ← Calendar ±14日（生データ）
  ├─ tasks[]            ← Tasks ±14日（生データ）
  └─ readingFB[]        ← 読書FB（ハイライト/評価サマリ）
  ※フル束を 1run=1doc でインライン保存（STEP1再実行時にGoogle API再叩き不要＝デモ再現性 I-14）
  ※サーバ専用書込・本人readのみ（Firestoreセキュリティルール §3）。生テキストのGCS分離・retention/TTLは将来F項目

personas/{personaId}  ← 作家＝人格データ（2種類）
  ├─ name, voiceStyle(文体軸), format(文章形式), persona(経歴+口癖+思想・リッチに), expertise[]
  ├─ pastBooks[]      ← このユーザーへの過去作（ファン化・続編の記憶）
  ├─ ephemeral: bool  ← true=テーマ確定時に都度生成した一時ペルソナ／false=永続（お気に入り等）
  └─ fromFavorite: bool ← true=favoriteAuthorsから再利用したペルソナ
  【一時ペルソナ】都度生成（ephemeral=true・テーマ確定のたびに各run5人 新規生成）
  【永続ペルソナ】①お気に入り登録済み（ephemeral=false） ②著者ペルソナ集.md の15件＝
                  固定プール選抜はせず、生成インスピレーション素材として任意参照（Firestore格納は任意）

users/{userId}
  ├─ profile: {                          ← STEP1（Pro）が生成・3層構造（2026-06-03 v2）
  │    base:            { industry, jobType, position, orgScale, readingGenres[] },  ← 安定・保持（再分析しない／orgScaleは初回Driveから一度抽出）
  │    currentWork:     { currentSituation, activeWorkThemes[], challenges[], upcomingKeyEvents[], evidence[] },  ← 週次・分析の主戦場（Drive/Calendar/Tasks）
  │    readingBehavior: { recentReads[], highlightsSummary, dropSignals[], feedbackSummary, serendipityTolerance, stylePreference },  ← 週次・学習シグナル（readingFB由来・初回空）
  │    analyzedAt
  │  }  ← currentWork/readingBehavior を週1（土朝）更新・1週間有効。base は更新頻度低
  ├─ initialProfile: {                  ← 登録時の初期入力（選択式・初回読者分析の主要シグナル）
  │    industry,            // ①業界（単一選択）
  │    jobType,             // ②職種（単一選択）
  │    position,            // ③役職（単一選択）
  │    recentInterests[],   // ④最近気になっていること（複数選択・最低1つ必須）
  │    readingGenres[],     // ⑤本のタイプ＝読み口・形態の好み（複数選択／著者ペルソナのformat/voiceStyle生成・許容度初期値に活用）※キー名は据え置き
  │    skipped              // スキップした場合 true（initialProfileなしでもDrive観測で動く）
  │  }
  ├─ favoriteAuthors: [                 ← 読書後にワンクリック保存（orphan防止で name/voiceStyle/format もコピー）
  │    { personaId, name, voiceStyle, format, savedAt }
  │  ]
  └─ connectedSources: {
       ── 確定（MVP3ソース・同一Google OAuth） ──────────────
       drive:    { enabled: true, folderIds[], labels[] }  ← Google Picker で**フォルダ単位**選択（G1-13＝MTG 2026-06-05確定。選択フォルダを folderIds[] でサーバ保持）
       calendar: { enabled: true, calendarIds[] }   ← 業務局面（calendar.readonly）
       tasks:    { enabled: true }                  ← 関心・課題（tasks.readonly。Keepの代替）
       readingFB:{ highlights[], logs[], simpleFB[] }
       ── 拡張候補（MVP対象外・後工程で採否決定） ──────────────
       gmail:    { enabled, subjectOnly: true }     ← PJ文脈（本文は読まない）
       slack:    { enabled, workspaceId }           ← チーム話題
       notion:   { enabled, pageIds[] }             ← 思考の現在地
       kindle:   { enabled, highlightsUrl }         ← 過去の刺さった知見
       pocket:   { enabled }                        ← 保存した記事
       youtube:  { enabled }                        ← 余暇・趣味の関心
       chatHistory: { enabled, exportPath }         ← 知識のギャップ（手動）
       podcast:  { enabled }                        ← 深く掘るジャンル
       browserHistory: { enabled }                  ← 無意識の関心（拡張経由）
     }
```

> **「企画(plan)」と「著者版(book)」を分けるのが今回の構成の肝。** 1つの承認企画から、著者A版・著者B版という複数の`books`が生まれ、`status=draft`で棚に並ぶ。ユーザーがどれかを選ぶと、その1冊だけ`reserved→writing→published`へ遷移。「どの著者版を選んだか」が最強の学習シグナルになる（構想6-1）。
> status遷移がUX・コスト・自律性を同時に解決。Firestoreのリアルタイムリスナーでフロントは状態変化を購読するだけ（ポーリング不要）。

## 4. スコープ再設計：死守 / 簡略 / 落とす（5週間の現実）

前回の積み上げ工数（実質6-7週相当）は5週に収まらない。**エージェントとフロントの2大工数を削るのが唯一の道。**

### 死守（無いと審査で戦えない＝MVP最小核）

| 要素 | 理由 |
|---|---|
| モードA自律企画（Cloud Scheduler起動） | 自律性の証拠＝基準1。技術的には軽い |
| **STEP2 企画チーム（3階層：リーダー／担当者／サブ＋スコアゲート）** | 基準1の本丸。**絶対削らない** |
| 書店UI（並ぶ・入荷理由・予約・読む の最小動線） | デモ主役＝基準4 |
| F3 入荷理由の可視化 | 解像度の証拠＝新規性 |
| 予約→1冊だけ後追い執筆 | 体験のコア＋コスト回避 |
| 最小DevOps（自動デプロイ＋Eval数件の品質ゲート） | 基準5・「まわす」 |

## 7. 必然性の実装：「実データ調査3観点＋2つの評価ループ」（基準1の生命線）

必然性の主張を、旧「多様性→敵対的選抜」から、**3階層の委任・評価構造＋編集ループ**に再構成する（2026-06-03 v2）。単一生成では原理的に到達できない3点で、マルチエージェントが演出でないことを示す。

**ロジック1：調査3観点＝サブが本当に違う仕事をする**
- 企画書フレーム（8項目）を埋めるために、観点の異なる調査サブを3つ置く：A読者局面／B市場・競合／Cテーマ知見。
- 特に**B・Cは Google検索 grounding で外部の実データを取得**し、単一LLMが内部知識だけでは原理的に出せない情報（今売れている本・最新の論点）を企画に注入する。
- これにより、同じテーマでも企画の中身が実データの分だけ実際に変わる（＝「出力が変わる最小構成」をクリア）。

**ロジック2：階層的スコア評価ゲート（企画の反復改善）**
- 企画リーダー（最上位）が、担当者の上げた企画書を**ルーブリックでスコア化**する。
- スコアが閾値未満なら `rejectionFeedback` を付けて担当者へ差し戻し、担当者は（既存のサブ成果を元に）練り直して再提出。**最高3ラウンド**で閾値到達 or 3ラウンド到達で承認する。
- 「自分で立てた企画を自分でスコアリングして閾値で突き返す」反復は、同一の単一生成では起こり得ない。**「必ず1回却下」のような演出ロジックは入れず、スコア閾値で自然に回るループとして実装**する。

**ロジック3：編集長⇄著者の編集ループ（成果物の反復改善）**
- STEP4（プレビュー・1R）とモードB（本文・最高3R）で、**編集長が著者の成果を採点し差し戻す**第2の評価ループを持つ。
- 企画（ロジック2）と執筆（ロジック3）で**役割の異なる2系統の評価ループ**が回る＝単一生成では原理的に再現できない多段の自己批正。

**「n=3サンプリングと同じでは？」への備え（重要）**
1. サブが**外部実データ（Google検索）**を取りに行く＝単一生成のサンプリングでは出せない情報を注入する
2. リーダーが**スコア閾値で差し戻す**評価ゲートを持つ（自己の出力を機械的に突き返せる）
3. **編集長が著者を採点して差し戻す**第2の評価ループ（企画とは別軸の品質ゲート）
4. 「3階層 vs 単一生成」をEval SetでA/B測定（将来検証）
- ⚠️ **「出版社という組織構造の再現」という比喩ではこの反論に答えられない。** 組織比喩は世界観・UX説明（5章側）に置き、基準1の主張には使わない。

**実装の最低ライン（5週間版）**: 「企画スコア化→差し戻し→再提出」と「編集長→著者差し戻し」の2ループがLangfuseのトレースに残ること（スコアとラウンド数が見える）。A/B測定は将来検証として割り切る。

## 8. Observability：GKEは捨ててマネージドへ

- **GKE（Google Kubernetes Engine）**= コンテナを大量に自動運用する重量級基盤。Kubernetes運用は専任インフラ前提で、1.5名・5週間には重すぎる。
- **Observability L4の本質は「GKEでLangfuseを運用すること」ではなく「Eval Setで品質をCIで継続評価する仕組みがあること」。** Langfuse Cloud（マネージドSaaS・無料枠）で運用ゼロのまま達成できる。
- マネージドを選ぶ判断自体が「身の丈に合った技術選定」として基準5で加点される。

## 9. Elasticsearch の必然性問題（スポンサー技術・5週間での判断）

- CLAUDE.mdは「ESでなければ成立しない必然性の論理説明」を求める。が、5週間では**動くこと優先で「外す勇気」も選択肢**。
- 入れる場合は一点に絞る: **「過去に出した本・既出概念・著者の作品履歴を横断的に意味検索して執筆時に参照する」**＝長文一貫性（構想6-3論点b）と続編の記憶。
- 「なぜES」への一行回答: **異種データ（本文・FB・設定）を単一の検索基盤で横断するから。**
- ⚠️ ただし続編記憶はFirestoreのpastBooks参照でも最低限は成立する。**ESの必然性を作り込む工数が、自律企画の作り込みを削るなら本末転倒。** W6の余力次第で判断（スポンサー点 < 動くデモ）。

## 10. DevOpsサイクル（基準5・「まわす」の証明）

```
コード/プロンプト/ペルソナ をGit管理
   │ push
   ▼
GitHub Actions（lint・Eval Set実行＝品質ゲート）
   │ pass
   ▼
Cloud Build → Cloud Run へ自動デプロイ
   │
   ▼
Langfuse がトレース収集 ＋ Eval結果を記録
   │
   ▼
フィードバック（読了率・FB）→ 次サイクルの企画方針へ（ループA）
```

- 核は **「Eval SetをCIの品質ゲートにする」**（L4）。例: 企画の関連性スコアが閾値未満ならデプロイを止める。
- 5週間版はEval **8件**で「ゲートが機能する形」を作れば十分（フル50件は将来）。
- Eval項目の設計は**鉄田担当**領域。
