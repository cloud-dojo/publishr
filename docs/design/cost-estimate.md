## 2. 1サイクル（1週間＝週15冊）の概算

> 構成: 土（観測+読者分析+本命企画→5冊）／水（本命企画→5冊）／日（セレンディピティ企画→5冊）＝**3run・15冊**。本文執筆は**予約された分のみ**（通常0〜1冊）。

> **2026-06-03 v2 反映**: モデル＝ハイブリッド（判断が重い工程Pro・**STEP1読者分析もProに上げ**）／企画＝調査3観点＋壁打ち／STEP4＝プレビュー編集ループ（1R）／モードB＝本文編集ループ（最高3R・弱い章のみ改稿・**予約上限 同時5冊**）。

| 工程 | モデル | 回数/週 | 1回の規模(概算・確証なし) | 支配度 |
|---|---|---|---|---|
| STEP1 読者分析 | **Pro** | 1 | 入力~20k(Drive抜粋10件+Cal/Tasks)・出力~2k（3層） | 小（週1のみ） |
| STEP2 調査サブ×3 | Flash(grounding) | 3run×3 | 各 入力~1k・出力~2k | 小（Flash） |
| STEP2 担当者（立案） | **Pro** | 3run×最大3R | 入力~5k・出力~2k | 中 |
| STEP2a リーダー採点 | **Pro** | 3run×最大3R | 入力~3k・出力~0.5k | 小〜中 |
| STEP3 キャスティング | **Pro** | 3run×1 | 入力~3k・出力~3k | 小 |
| STEP4 プレビュー編集（著者+編集長×5著者×1R） | **Pro** | 3run×5×(執筆+採点+1R改稿) | 各 入力~3k・出力~2k | 中（短文・1Rなので限定的） |
| STEP5 表紙 Imagen | Imagen | **15枚** | $0.02〜0.04/枚 | 中（先行生成） |
| モードB 本文編集ループ | **Pro** | 予約分のみ(0〜5冊) | 著者出力~30〜50k×(初稿+弱章改稿) ＋編集長採点 | **1冊あたり最大・同時上限5冊が天井** |

### 概算レンジ（確証なし）
- **企画(Pro担当者/リーダー＋Flash調査)**: 週 ~$0.2〜0.4。
- **STEP4 プレビュー編集(Pro・5著者×1R×3run)**: 週 ~$0.1〜0.7（短文・1Rのため限定的）。
- **STEP5 表紙 Imagen 15枚**: 週 ~$0.3〜0.6。
- **モードB 本文編集ループ(Pro・100p・最高3R・弱章のみ改稿)**: 1冊あたり ~$0.4〜0.9。**予約上限 同時5冊なら最悪 +$2〜4.5**。
- **→ 1サイクル合計（予約1冊想定・デモ）**: ざっくり **$2〜3.5/週**（確証なし）。**同時5冊予約の最悪ケースで週$4.5〜7程度**。

> 桁感の結論: **Proベース化＋編集ループでも、予約上限 同時5冊が天井になり定常運用は安い**。問題は依然「**開発中の反復実行**」。予約上限を3→5に上げたぶん最悪ケースは増えるが、デモは1冊なので影響は限定的。

## 3. 開発反復での枯渇リスク（ここが本題）

¥10,000 ≈ ~$66（確証なし）。1サイクル~$2〜3.5なら十数〜数十回は回せるが、**開発中は以下が膨らむ**：

1. **モードB本文編集ループ(Pro・最高3R)**: プロンプト調整で100p生成を何度も回すと、1回$0.4〜0.9×3R×試行回数。**最大の枯渇要因**。→ dev時は本文短縮＋ラウンド1Rで抑える。
2. **STEP4 プレビュー編集(Pro・5著者)**: 1Rでも5著者×3runで効く。dev時は著者数を絞る。
3. **表紙 Imagen の作り直し**: UI/装丁調整で15枚×N回。→ devモック。
4. **Eval(LLM-as-judge=Pro)のCI実行**: pushのたびに8件採点。→ 発火条件を絞る。
5. **grounding課金**（確証なし）: 調査サブB/Cの多用で未知の上振れ。

## 4. 開発時の上限ガード（設計に織り込む）

| ガード | 内容 | 効果 |
|---|---|---|
| **本文ページ数のdev切替** | 環境変数で本文を「dev=3〜5p / prod=100p」に切替。プロンプト検証はdevで | Pro出力コストを1〜2桁削減 |
| **Imagenのdevモック** | dev中は表紙をプレースホルダ画像 or 生成スキップ。本番/録画時のみ実生成 | Imagen費を実質ゼロに |
| **Eval CIの発火条件を絞る** | プロンプト/ペルソナ/企画ロジック変更時のみEval実行（全push禁止）。MVPスコープ §9-5の方針を厳守 | judge(Pro)呼び出し回数を抑制 |
| **冊数のdev削減** | dev中は1run=5冊→1〜2冊に | 全工程を比例削減 |
| **予約上限 同時5冊（本番ルール）** | モードB本文の同時生成を最大5冊に制限（reserve APIでチェック） | モードBコストの天井 |
| **編集ラウンドのdev削減** | dev中は STEP4=1R / モードB=1R に固定（本番はモードB最高3R） | Pro出力を1/3に |
| **ラウンド上限の遵守** | STEP2＝最高3R／STEP4＝1R／モードB＝最高3R。無限ループ防止はescalate/max_iterationsで担保（W1実証・台帳ADK論点2） | 暴走防止 |
| **予算アラートの活用** | ¥10,000の50%/90%/100%（設定済）。50%で一度コスト内訳をLangfuseで点検 | 早期検知 |

> 推奨: 上記の「本文ページ数」「Imagen実行」「冊数」を**1つの dev/prod フラグ**で束ね、デフォルト dev にしておく。録画・本番デモ時のみ prod に上げる。
